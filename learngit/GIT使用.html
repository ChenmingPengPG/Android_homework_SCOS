使用SSH连接

1、在用户主目录下

ssh-keygen -t rsa -C "youremail@example.com"

此时用户主目录下有 .ssh文件夹，里面有id_rsa 和 id_rsa.pub 分别为私钥和公钥

2、登录github，在account settings 下 add ssh key

key文本框里添加id_rsa.pub文件的内容

3.gitbub下创建一个repository， 比如learngit

4.在本地仓库下 

xxxxxxxxxx
git remote add origin git@github.com:(账户名)/(repository名).git

origin 则是本机上给远程库的名字

5 

xxxxxxxxxx
git push -u origin master

第一次使用-u参数，会把本地本地的master分支内容推送的远程新的master分支，还会把本地的master分支和远程的master分支关联起来，在以后的推送或者拉取时就可以简化命令。

6.之后可以通过

xxxxxxxxxx
git push origin master

创建与合并分支

严格来说head指向master，master指向提交，所以head指向的就是当前分支。

创建新的分支时，例如dev时，git新建了一个指针交dev，指向master的相同提交，再把head指向dev上

 

从现在开始，对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev往前移动一步，而master指针不变

如果在dev的工作完成了，就可以把dev合并到master上。（将master指向dev）

合并完分支后，甚至可以删除dev分支。删除dev分支就是把dev指针给删掉，删掉后，我们就剩下了一条master分支：

xxxxxxxxxx
git checkout -b dev

-b 表示创建并切换

相当于

xxxxxxxxxx
git branch dev
git checkout dev

然后使用

xxxxxxxxxx
git branch

查看当前的分支，当前分之前有*号

然后便可以在当前分支上做修改

切换回master

xxxxxxxxxx
git checkout master

将dev分支工作合并到master上

xxxxxxxxxx
git merge dev

删除dev分支

xxxxxxxxxx
git branch -d dev

采用分支工作更安全

解决冲突

创建新的分支

xxxxxxxxxx
git checkout -b feature1

做出修改,commit后，再切换到master

xxxxxxxxxx
git checkout master

做出不同修改,commit后

xxxxxxxxxx
git merge feature1

会提示有文件发生冲突

可以用status查看冲突文件

然后手动解决冲突，再提交

xxxxxxxxxx
git add readme.txt
git commit -m "conflict fixed"

可以用以下命令查看分支的合并情况

xxxxxxxxxx
git log --graph --pretty=oneline --abbrev-commit

最后删除feature1分支

xxxxxxxxxx
git branch -d feature1

分支管理策略

合并分支时，如果可能git使用fast forward模式，这种模式下删除分支后，会丢掉分支信息

可以使用

xxxxxxxxxx
git merge --no-ff -m "merge with no-ff" dev

来禁止fastforward模式

在实际开发中，我们应该按照几个基本原则进行分支管理：

首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；

那在哪干活呢？干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；

你和你的小伙伴们每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了。

stash 可以把当前工作现场储藏起来，等以后恢复现场继续工作

xxxxxxxxxx
